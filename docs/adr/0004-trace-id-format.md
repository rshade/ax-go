# ADR-0004: Trace ID Format

## Status

ACCEPTED — 2026-05-28.

## Context

Distributed tracing requires unique IDs that interop with the broader
observability ecosystem (Tempo, Jaeger, Honeycomb, Datadog). An agent
orchestrator running an ax-go CLI may already have a trace open and pass
it via the W3C `TRACEPARENT` environment variable.

The format choice is effectively constrained by what tracing backends
accept and what the OpenTelemetry SDK generates.

## Decision Drivers

- Compatibility with W3C Trace Context (the lingua franca of distributed
  tracing).
- Zero-friction integration with OTel-compatible backends.
- Avoid implementing ID generation if a battle-tested library already
  does it.
- IDs appear in logs, error envelopes, and stdout metadata; the format
  must be ASCII-safe and easy to grep.

## Considered Options

### A. W3C Trace Context via OTel SDK (16-byte trace_id, 8-byte span_id, hex-encoded)

Generated by the OpenTelemetry SDK's default `IdGenerator`. Zero
implementation cost. Direct interop with every tracing backend.

### B. UUID (v4 or v7) with hyphens stripped to fit 16-byte hex

Familiar format. Requires conversion at the OTel boundary. No
observability-backend benefit over A.

### C. ULID

Time-sortable. Requires Base32→hex conversion at the OTel boundary. No
observability-backend benefit over A.

## Decision

Adopt **Option A**: W3C Trace Context format, generated by OTel's native
`IdGenerator`. `trace_id` is a 32-character lowercase hex string;
`span_id` is a 16-character lowercase hex string.

Rationale: zero implementation cost, perfect interop with the OTel
ecosystem, and the W3C standard is the path of least resistance for every
downstream tool that will consume ax-go's traces.

UUIDs and ULIDs remain valid for **non-trace** identifiers (resource IDs,
idempotency keys, run IDs) — see ADR-0007 for the broader ID strategy.

## Consequences

- ax-go takes a direct dependency on `go.opentelemetry.io/otel/sdk`.
- The `TRACEPARENT` env var is the canonical inbound trace-context channel
  (see [`specs/004-real-otel-export`](../../specs/004-real-otel-export/)
  for the real OTel export lifecycle).
- All log lines, error envelopes (ADR-0002), and stdout JSON metadata
  blocks carry `trace_id` (and `span_id` where appropriate) sourced from
  the active OTel span context.
- When no span is active, fields carry zero-value valid hex strings so
  consumer parsers do not branch on absence.

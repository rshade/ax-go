# ADR-0007: ID Strategy Across the Library

## Status

ACCEPTED — 2026-05-28.

## Context

Three distinct ID needs surface in ax-go:

1. **Observability IDs:** `trace_id`, `span_id` (correlate execution).
2. **Idempotency keys:** prevent duplicate-create from agent retries.
3. **Resource / entity IDs:** identify a newly-created thing (a record,
   a job, a temp file).

Each has different requirements (interop, sortability, security). Mixing
them creates subtle bugs — using a `trace_id` as a resource ID breaks
observability when the same resource is referenced from multiple traces.

## Decision Drivers

- Observability IDs must follow the W3C standard (decided in ADR-0004).
- Idempotency keys must be unique enough to detect duplicates over the
  agent's retry window (minutes to hours).
- Entity IDs benefit from time-sortability for some workloads (logs,
  audit trails); not all.
- Avoid unnecessary dependencies — one well-chosen library is better than
  three.

## Categories and the rule

### Observability (`trace_id`, `span_id`)

**Use OTel-generated W3C IDs** (see ADR-0004). Never substitute. Surface
via the ZeroLog hook (ADR-0005), error envelopes (ADR-0002), and stdout
JSON metadata blocks.

### Idempotency keys (`--idempotency-key`)

When auto-generated by the base: **UUID v4** (random, no temporal info
leak). User-supplied keys are treated as opaque strings of any shape.
Surfaced in output envelopes for agent correlation.

### Entity IDs (new resources, jobs, temp files)

Two viable candidates:

#### A. UUID v7

128-bit; first 48 bits are Unix-time-millis (sortable); remaining 74 bits
random; standardized via RFC 9562. Backwards-compatible with v4 for
systems that just need uniqueness.

Pros: time-sortable; widely supported; same library covers v4 for
idempotency.
Cons: leaks creation time; longer string representation (36 chars).

#### B. ULID

128-bit; 48 bits Unix-time-millis (sortable); 80 bits random; Crockford
Base32 encoded (26 chars vs. UUID's 36).

Pros: shorter; time-sortable; URL-safe.
Cons: separate library from the UUID generator; less standardized than
UUID v7.

## Decision

Entity IDs use **UUID v7** via `github.com/google/uuid` (which covers
both v4 and v7 in a single dependency). Idempotency keys auto-generated
by the base are **UUID v4** from the same library. Observability IDs
are W3C via the OTel SDK (ADR-0004) and are never substituted.

ULID is skipped — UUID v7 covers time-sortability via the RFC 9562
standard with no additional dependency, and standardization beats the
slightly-shorter ULID string representation for our use case.

## Consequences

- A single canonical library handles UUID generation
  (`github.com/google/uuid` covers v4 and v7 in current versions).
- ULID would require an additional dependency
  (`github.com/oklog/ulid/v2`).
- The base package exposes a tiny `ax.NewEntityID()` helper so the
  choice can be revisited without touching every call site.
- Trace IDs and entity IDs must never appear in the same column / field
  in a log line — they have different meanings and grep behavior.

# Data Model: Cut v0.1.0 — Make ax-go a Consumable, Pinnable Release

**Feature**: `003-cut-v010-release` | **Date**: 2026-06-09

This feature creates no new runtime data structures. Its entities are (a) two
existing public wire shapes being *frozen*, and (b) release-process artifacts.
Shapes below are documented as they exist in `json.go` — any change to them is
explicitly out of scope (spec Assumptions).

## Wire Entities (frozen, not modified)

### Success Envelope — `ax.Envelope[T]`

The standard bounded JSON success payload for every ax-go-based CLI.

| Field | Go type | JSON key | Presence | Determinism |
|---|---|---|---|---|
| `Data` | `T` (generic) | `data` | always | caller-controlled |
| `Meta` | `Metadata` | `meta` | always | see Metadata |

**Validation rules** (contract, enforced by golden test):

- Serialized by `WriteJSON` as strict, minified JSON + single trailing `\n`.
- Field order is fixed by struct declaration order: `data`, then `meta`.
- `Envelope[T]` carries **no `version` field** (spec Resolved Decision 4).

### Envelope Metadata — `ax.Metadata`

| Field | Go type | JSON key | Presence | Determinism |
|---|---|---|---|---|
| `TraceID` | `string` | `trace_id` | always (zero value `ZeroTraceID` when no span) | non-deterministic → pinned via OTel `ContextWithSpanContext` |
| `SpanID` | `string` | `span_id` | `omitempty` | non-deterministic → pinned via same seam |
| `IdempotencyKey` | `string` | `idempotency_key` | `omitempty` | non-deterministic (auto UUID v4) → pinned via `ax.WithIdempotencyKey` |
| `DryRun` | `bool` | `dry_run` | `omitempty` | deterministic (context flag) |

**State note**: because `span_id` and `idempotency_key` are `omitempty`, the
golden fixture MUST be generated from a context that populates them (research
D4) — otherwise those keys are absent from the fixture and unguarded.

### NDJSON Line

One newline-delimited JSON record emitted by `WriteJSONLine` for streaming
output. Wire-identical today to a success envelope serialized as one line
(implementation delegates to `WriteJSON`), but contractually independent —
hence its own fixture (research D5).

## Fixture Entities (new files)

| Fixture | Locks | Generated by | Exercised by |
|---|---|---|---|
| `testdata/success_envelope.golden.json` | `Envelope[T]` + `Metadata` success shape via `NewEnvelope` → `WriteJSON` | fixed-context construction (research D4) | new golden test in `json_test.go` |
| `testdata/ndjson_line.golden.json` | NDJSON line shape via `WriteJSONLine` | same technique, distinct payload | new golden test in `json_test.go` |

**Lifecycle / state transitions** (both fixtures):

```text
ABSENT ──(test written first; run fails at os.ReadFile — correct TDD failure)──►
WRITTEN ──(test passes byte-for-byte)──► LOCKED (any drift fails CI; FR-006)
```

Pinned input values are defined normatively in `contracts/` so test and fixture
cannot drift independently.

## Release-Process Entities

| Entity | Representation | Created by | Constraints |
|---|---|---|---|
| Release PR | GitHub PR from release-please | release-please-action on push to `main` | MUST propose `0.1.0` — guaranteed by `Release-As: 0.1.0` footer (research D1); MUST contain generated `CHANGELOG.md` section only (FR-011) |
| Release Tag | git tag `v0.1.0` on `main` | release-please on release-PR merge | immutable once the module proxy caches it; `make ci` MUST be green on the tagged commit (SC-006) |
| `CHANGELOG.md` | generated file | release-please (`changelog-sections` mapping) | zero hand-authored lines (SC-007) |
| Workflow run | `.github/workflows/release-please.yml` | push to `main` | authenticated by `GITHUB_TOKEN`; green on the release-PR merge (SC-004) |
